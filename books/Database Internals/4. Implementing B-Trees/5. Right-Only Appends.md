In monotonically increasing indexes (like integer primary keys) the behaviour effectively becomes a linkedlist where all the writes happen to the right-most node.  
In practice, each new insert is a new ID that is greater than the previous one so we keep inserting into the same page until the page is full (like appending to a log file).  
When a new insert comes in to a full page we create a new page and store the value in there.  
Postgres (fastpath) tracks the last page and thus skips having to read and traverse all the previous pages to insert the new value.  
SQL (quickbalance) creates a new page with a pointer to its parent instead of rebalancing the tree.  
In other words they do the same thing but optimise away either traversal (psql) or rebalancing (sql).  
# Bulk Loading
If you sort your data by the keys you may build the tree from the bottom up in page increments instead of inserting each key one at a time and appending to the right until we need to create a new page.  
When one page is full you emit the separator keys for the parent and create child pointers. Do this for each page at the bottom of the tree and for each parent to efficiently recreate the tree by avoiding any splits and merges.
