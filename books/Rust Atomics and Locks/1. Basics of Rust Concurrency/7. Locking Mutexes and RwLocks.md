The job of a mutex is to ensure threads have exclusive access to some data by temporarily blocking other threads that try to access it at the same time.
Locking a mutex prevents other threads from using it by putting subsequent lock requests to sleep until the lock holder unlocks the mutex.
# Rust's Mutex
Other threads cannot unlock the mutex because the thread that set the lock is given a MutexGuard which is an exclusive reference.
Once the guard is dropped the Drop trait unlocks the mutex allowing other threads to set the lock.
Locks should be set for critical sections and dropped as soon as possible to prevent other threads from being blocked.
# Lock Poisoning
If a thread that holds a lock panics that lock is said to be poisoned. The lock is dropped and the Mutex is unlocked but subsequent calls to `.lock()` still lock the mutex but also return an `Err` which contains the `MutexGuard` to allow the caller to clean up any inconsistent state the `Mutex` may have been left in.
`MutexGuard` lives until the end of its statement, so in constructs like `if let` it can keep a lock held longer than expected when values are borrowed. Splitting the lock into a separate `let` forces an earlier drop and releases the lock sooner.
# Reader-Writer Lock
Has 3 states
- Unlocked
- Exclusive lock (1 writer)
- Shared lock (many readers)
It's a more complex version of a `Mutex` splitting the api into reading and writing instead of a single `lock()` method.
It acts as a multi-threaded version of `RefCell`.
A `Mutex` requires `T: Send` while `RwLock` required `T: Send + Sync`.
Rust provides 1 general purpose implementation of `RwLock` while other variations exist.
